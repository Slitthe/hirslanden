# Type-safe i18n for `skin-cancer-screening`

**Date:** 2026-06-24
**Status:** Approved (design)
**Scope:** `apps/skin-cancer-screening`

## Goal

Add internationalisation to the skin-cancer-screening app with full TypeScript
type-safety, supporting German (`de`), French (`fr`), and English (`en`).

### Requirements (verbatim, from the request)

1. Used via `<TranslationProvider locale={locale}>`.
2. Consumed via a `useTranslation` hook that returns a `translate` function:
   `const { translate } = useTranslation()`, then `translate("general.buttons.ok")`
   and `translate("general.someInterpolatedValue", { value: "42" })`.
3. Fully type-safe:
   - a. A key that does not exist emits a TypeScript error.
   - b. A test asserts all languages contain the same keys.
   - c. Interpolation params are type-safe: a key with interpolation errors if the
     required params are not provided.
4. Interpolation syntax is `{{myType}}`.
5. Translations live in `src/locale/[locale].json`.
6. Languages: DE + FR + EN.

### Decisions made during brainstorming

- **Canonical/base locale: `en`.** `en.json` is the source of truth for generated
  types and the runtime fallback locale. (All three locales must still contain
  identical keys — enforced by a test — so this only selects which file drives type
  generation and what we fall back to.)
- **Type generation: committed + test-enforced.** A `pnpm gen:i18n` script writes a
  committed `generated.ts`; a vitest test regenerates in-memory and fails if it is
  stale. This works with the repo's plain `tsc --noEmit` `typecheck` task and CI; no
  hidden bundler magic.
- **Param value type: `string | number`** (runtime stringifies via `String(v)`), so
  both `{ value: "42" }` and `{ value: 42 }` type-check.

## Key technical constraint (the reason for codegen)

TypeScript **widens JSON string values to `string`** on import (verified empirically:
assigning an arbitrary string to `typeof en['a']['b']` compiles). Object *keys* stay
literal, so key-existence safety (3a) is free from a JSON import — but the `{{var}}`
placeholders inside *values* are erased at the type level, so typed interpolation (3c)
**cannot** be derived from a JSON import alone. We therefore generate a small types
artifact from the source JSON.

The generated artifact is a precomputed interface (not type-level recursion over an
`as const` mirror): the flattening + `{{var}}` extraction happen once in the codegen
(plain JS regex), keeping consumer/IDE type-checking fast and the generated output
human-readable.

## Architecture

- **Content:** hand-edited `src/locale/{en,de,fr}.json`. `en` is the source of truth.
- **Codegen:** `scripts/gen-i18n.mjs` reads `en.json`, flattens to dot-path keys,
  extracts `{{tokens}}` per value, and emits `src/i18n/generated.ts`.
- **Compile-time safety (3a + 3c):** flows entirely from the generated `Translations`
  interface.
- **Runtime:** all three JSONs are statically imported and bundled (tiny, synchronous,
  no Suspense / no async loading). The provider selects the active locale by prop;
  `en` is the fallback.

### File layout

```
apps/skin-cancer-screening/

  src/

      bundles.ts                # static JSON imports, flatten(), BUNDLES, Locale, FALLBACK_LOCALE
```

## Components and contracts

### `generated.ts` (generated, committed)

Maps every flattened key to its params object type. No-param keys use
`Record<never, never>` (so `keyof` is `never`, and biome's `noBannedTypes` is not
triggered as it would be by `{}`):

```ts
// AUTO-GENERATED by scripts/gen-i18n.mjs from src/locale/en.json. Do not edit.
// Run `pnpm gen:i18n` to regenerate.
export interface Translations {
  'general.buttons.ok': Record<never, never>
  'general.someInterpolatedValue': { value: string | number }
  // ...one entry per flattened key
}
```

### `bundles.ts`

```ts
import de from '../locale/de.json'
import en from '../locale/en.json'
import fr from '../locale/fr.json'

// Local helper: recursively turns a nested object of string leaves into
// Record<dotPath, string>. Defined inline in this module (the codegen has its
// own flattenKeys in scripts/gen-i18n.mjs since .mjs cannot import src/ at runtime).
function flatten(obj: unknown, prefix = ''): Record<string, string> { /* ... */ }

export const BUNDLES = {
  de: flatten(de),
  en: flatten(en),
  fr: flatten(fr),
} satisfies Record<string, Record<string, string>>

export type Locale = keyof typeof BUNDLES        // 'de' | 'en' | 'fr'
export const FALLBACK_LOCALE: Locale = 'en'
```

### `context.ts`

```ts
import { createContext } from 'preact'
import type { Locale } from './bundles'

export interface TranslationContextValue {
  locale: Locale
  messages: Record<string, string> // flattened active locale
}
export const TranslationContext = createContext<TranslationContextValue | null>(null)
```

### `TranslationProvider.tsx`

```tsx
export function TranslationProvider({
  locale,
  children,
}: { locale: Locale; children?: ComponentChildren }) {
  const value = useMemo(() => ({ locale, messages: BUNDLES[locale] }), [locale])
  return <TranslationContext.Provider value={value}>{children}</TranslationContext.Provider>
}
```

### `interpolate.ts`

```ts
const TOKEN = /\{\{\s*(\w+)\s*\}\}/g
export function interpolate(template: string, params?: Record<string, string | number>): string {
  if (!params) return template
  return template.replace(TOKEN, (_m, name) =>
    params[name] === undefined ? `{{${name}}}` : String(params[name]),
  )
}
```

### `useTranslation.ts`

```ts
import type { Translations } from './generated'
export type TranslationKey = keyof Translations

type ParamsArg<K extends TranslationKey> =
  [keyof Translations[K]] extends [never]
    ? []                            // no interpolation params -> 2nd arg forbidden
    : [params: Translations[K]]     // params required and typed

export type Translate = <K extends TranslationKey>(key: K, ...args: ParamsArg<K>) => string
export interface Translator { translate: Translate }

// Used as: const { translate } = useTranslation()
export function useTranslation(): Translator {
  const ctx = useContext(TranslationContext)
  if (!ctx) throw new Error('useTranslation must be used within a <TranslationProvider>')
  const { messages } = ctx
  const translate: Translate = (key, ...args) => {
    const template = messages[key] ?? BUNDLES[FALLBACK_LOCALE][key] ?? key
    return interpolate(template, args[0] as Record<string, string | number> | undefined)
  }
  return { translate }
}
```

The non-distributive `[keyof T] extends [never]` check is required (`never` in a bare
conditional distributes to `never`).

### `index.ts` (public API)

```ts
export { TranslationProvider } from './TranslationProvider'
export { useTranslation, type TranslationKey } from './useTranslation'
export type { Locale } from './bundles'
```

## Codegen (`scripts/gen-i18n.mjs`)

- Exports a pure generator: `generateTypesSource(json: object): string` and a
  `flattenKeys(json)` helper.
- When run as the main module (`pnpm gen:i18n`): read `src/locale/en.json`, produce
  the source string, write `src/i18n/generated.ts`.
- The vitest freshness test imports `generateTypesSource` from this same module and
  compares its output to the committed file (newline-normalized) — single source of
  truth for both the CLI and the test.
- Token extraction: `{{name}}` via `/\{\{\s*(\w+)\s*\}\}/g`, de-duplicated, in
  first-seen order; emits `Record<never, never>` when a value has no tokens.

## Tests

Runtime (vitest, `src/i18n/i18n.test.tsx`):

1. **Locale key parity (3b):** flatten `de`/`en`/`fr`, assert identical sorted key sets.
2. **Placeholder parity (bonus):** for each key, assert the set of `{{tokens}}` is
   identical across all three locales.
3. **Generated freshness (anti-drift):** assert `generateTypesSource(enJson)` equals
   the committed `generated.ts`; failure message instructs to run `pnpm gen:i18n`.
4. **Behavior:** render the provider at each locale; assert translated and interpolated
   output, locale switching via the `locale` prop, and `en`/raw-key fallback. Also
   assert the hook throws outside a provider.

Compile-time (`src/i18n/types.assert.ts`, validated by the existing `pnpm typecheck`):

- `@ts-expect-error` on an unknown key (3a).
- `@ts-expect-error` on a params-requiring key called with no params (3c).
- `@ts-expect-error` on a wrong/extra param shape.
- Positive cases (valid key, valid params) compile with no directive.

## Out of scope (YAGNI)

- Pluralization, gender, or ICU message syntax.
- Nested arrays / non-string leaves.
- Async / lazy locale loading or code-splitting per locale.
- A `t()`-function escape hatch or dynamic (non-literal) keys — the spec's
  `useTranslation` hook is the sole public API.
- Locale detection / persistence — the `locale` is supplied by the host via the prop.

## Adversarial review — findings & resolutions

A multi-agent review (5 dimensions, each finding independently verified) ran after
implementation. Confirmed findings and their disposition:

Fixed:
- **interpolate used `name in params`** — matched inherited `Object.prototype` members
  (`{{toString}}` etc.) and leaked function source. Now uses `Object.hasOwn`. Regression
  test added.
- **Biome could reformat `generated.ts`** (strip quotes from top-level identifier keys;
  wrap >100-char lines), which would break the freshness test once a non-dotted or
  many-token key is added. `generated.ts` is now biome-ignored (`files.includes` in
  `biome.json`); `tsc` still type-checks it and the freshness test still guards content.
- **Codegen could emit invalid TS** for digit-leading tokens (`{{1st}}`) or keys with
  quotes/backslashes. The generator now emits property names via an `asNeeded`-style
  helper (bare identifier, else escaped single-quoted literal). Tests added.

Documented as a known limitation (out of scope of the stated requirements):
- **Union-typed keys drop the required-params guarantee (3c).** For a key typed as a
  *union* of literals (e.g. chosen via a ternary), `ParamsArg<K>` collapses to `[]`, so
  a missing-params mistake is not caught. Every literal call site (requirement 2's
  documented usage) is fully guarded. A clean *sound* fix requires added type
  complexity, so the limitation is documented in `useTranslation`'s JSDoc and
  `types.assert.ts` rather than fixed. Can be hardened on request.

Accepted as nits (no change): `flatten` coerces non-string leaves / descends arrays
(current locale data is all strings); the freshness test resolves `generated.ts` via
`process.cwd()` (correct for the turbo / pnpm-filter invocation used here); the
hand-written `gen-i18n.d.mts` is not cross-checked against the `.mjs` body.
